# Trigger Allocation Commit Helper Spec

Status: Implemented in M10-108.

## Purpose

Commit trigger allocation modifiers into a `CombatModifierLedger` clone using
the modifier ledger produced by `TriggerCheckResolutionBundle`.

This is still a pure helper. It does not mutate live `GameState`.

## API

```csharp
TriggerAllocationCommitResult TriggerAllocationCommitHelper.Commit(
    CombatModifierLedger currentLedger,
    TriggerCheckResolutionBundle bundle)
```

## Inputs

- current combat modifier ledger, or null for empty,
- accepted trigger check resolution bundle,
- bundle modifier ledger generated by trigger allocation adapter.

## Result Metadata

`TriggerAllocationCommitResult` contains:

- `accepted`
- `rejection_reason`
- `source_id`
- `check_source`
- `player_index`
- `check_index`
- `checked_card_id`
- `trigger_type`
- `appended_modifier_count`
- `ledger`

## Rejections

- `TRIGGER_ALLOCATION_COMMIT_BUNDLE_MISSING`
- `TRIGGER_ALLOCATION_COMMIT_BUNDLE_REJECTED` or the bundle rejection reason
- `TRIGGER_ALLOCATION_COMMIT_MODIFIER_LEDGER_MISSING`

## Safety Rules

- The helper returns a new ledger clone.
- The helper must not mutate the source ledger.
- The helper must not mutate the source bundle.
- The helper must not mutate `GameState`.
- The helper must not append `GameState.event_log`.
- The helper must not publish network payloads.
- If source metadata is already masked, the committed ledger/event metadata must
  remain masked.

## Current Boundary

M10-108 commits allocation modifiers into a ledger clone only. M10-109 wires
cleanup timing integration. Live combat stat application still goes through
existing projection/scaffold paths.

## Verification

EditMode coverage verifies:

- accepted modifier append,
- empty accepted modifier ledger,
- missing/rejected/missing-ledger rejections,
- input ledger and bundle no-mutation,
- masked trigger source metadata preservation.
